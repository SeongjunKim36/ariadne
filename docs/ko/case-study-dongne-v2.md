# Ariadne를 `dongne-v2` 실계정에 적용해본 기록

> 상태: 초안 · 작성일: 2026-04-16

## 한 줄 요약

Ariadne를 `dongne-v2` 실계정에 연결했을 때, 단순 리소스 목록이 아니라 **관계가 있는 인프라 그래프**, **감사 결과**, **질의 응답**, **타임라인/드리프트 기반 변경 추적**까지 한 흐름으로 확인할 수 있었다.

## 왜 이 검증을 했나

LocalStack과 단위 테스트만으로는 “실제 계정에서도 쓸 수 있는가?”를 충분히 설명하기 어렵다. 이 문서는 Ariadne가 실계정에서 어느 정도까지 바로 동작했고, 어떤 질문에 답할 수 있었는지 보여주기 위한 케이스 스터디 초안이다.

## 검증 범위

- 실계정 인증: `AWS_PROFILE=ariadne`
- 핵심 수집기: VPC, Subnet, SG, EC2, RDS, ALB, ECS, Lambda, S3, Route53, IAM Role
- 감사 엔진
- NL Query
- Snapshot / Diff / Timeline / Terraform Drift

## 확인한 것

### 1. 실제 계정 스캔

실계정 인증이 된 상태에서 Ariadne 백엔드는 계정 정보를 정상 인식했고, 스캔이 완료되면 그래프 API와 Topology UI에서 리소스 간 관계를 확인할 수 있었다.

이미 문서화된 실검증 기준:

- Phase 1: `22 nodes / 21 edges`
- Phase 2 상세 수집 재검증: `24 nodes / 45 edges`
- Phase 5 릴리즈 재검증: `24 nodes / 45 edges`, `snapshot 2개`, `latest diff 0 changes`

이 숫자는 단순히 리소스 개수만 의미하지 않는다. 보안 그룹 상세 엣지(`ALLOWS_TO`, `EGRESS_TO`), 추론 관계(`LIKELY_USES`), 서비스 연결 관계 등이 포함되면서 “운영자가 읽을 수 있는 구조”가 되었다.

### 2. 실제 질문에 답할 수 있었는가

Phase 3 시점 검증 기준으로, 다음 유형의 질문은 실계정 데이터에서 안전하게 처리되었다.

- “0.0.0.0/0으로 열린 SG 중 위험한 거 보여줘”
- “이 RDS를 쓰는 애들이 누구야?”
- “prod vs staging 차이가 뭐야?”

특히 마지막 질문은 환경 태그가 없는 경우에도, 근거 없는 답을 꾸며내지 않고 **비교에 필요한 환경 정보가 부족하다고 설명하는 쪽으로 fail-safe**하게 동작했다.

### 3. Phase 4로 무엇이 더 좋아졌나

Phase 4를 지나면서 Ariadne는 “지금 그래프”에서 멈추지 않고, 아래 질문까지 커버하게 됐다.

- “어제 대비 오늘 뭐가 바뀌었어?”
- “Terraform state와 실제 AWS가 어디서 어긋나?”
- “EventBridge로 들어온 중요한 변경 이후에 어떤 부분 재수집이 일어났어?”

이 변화는 운영 흐름에서 중요하다. 사람은 결국 현재 상태보다 **변경의 방향과 영향**을 더 자주 궁금해하기 때문이다.

### 4. 실제 감사 결과는 어땠나

Phase 5 릴리즈 재검증 시점에 감사 엔진을 다시 실행했을 때, 실계정에서는 총 `62 findings`가 나왔다.

- HIGH: `27`
- MEDIUM: `14`
- LOW: `21`

눈에 띈 결과는 퍼블릭으로 열린 Security Group, 퍼블릭 IP와 넓은 규칙이 동시에 붙은 EC2, 그리고 SSL 강제가 빠진 S3 버킷이었다. 이건 “도구가 돌아간다”를 넘어, 실제로 운영자가 봐야 할 리스크를 바로 꺼내준다는 의미가 있다.

## 좋았던 점

### 그래프가 운영자의 시선을 따른다

AWS 콘솔은 개별 서비스 화면으로 나뉘어 있고, Cartography나 Resource Explorer는 수집/조회는 강하지만 “운영자가 한눈에 이해하는 UX”는 별도로 만들어야 한다. Ariadne는 최소한 다음 흐름을 한 화면 안에서 이어준다.

- 스캔
- 토폴로지 확인
- 상세 패널 열기
- 감사 결과 확인
- 질의 실행
- 변경(diff) 보기

### 보안/LLM 경계가 분리돼 있다

NL Query나 요약 기능이 들어간 프로젝트는 “편한 대신 위험한” 경우가 많다. Ariadne는 리댁션 엔진, allowlist, transmission level, audit log를 분리해서 이 부분을 구조적으로 다루려는 점이 장점이다.

### 문서와 구현이 꽤 가까워졌다

Phase 4 종료 시점에는 체크리스트와 실제 구현 차이가 많이 줄었다. 공개 저장소로 운영할 때 이 부분은 신뢰도에 직접 영향을 준다.

## 아직 남은 점

### 대규모 계정 성능

현재 구조는 중소 규모 계정에는 충분히 설명 가능하지만, 수백~수천 리소스 계정에서 React Flow 렌더링과 Neo4j 쿼리 비용을 더 체계적으로 검증해야 한다.

### 데모 자산

README와 케이스 스터디는 갖춰졌고, Topology / Audit / Timeline 스크린샷도 생성했다. 다만 GIF/영상 데모 자산은 아직 더 보강할 여지가 있다.

### 운영 배포

로컬 개발 환경과 실계정 검증은 끝났지만, 외부에 보여줄 안정적인 데모 인스턴스 운영은 아직 다음 단계다.

## 포트폴리오 메시지

이 프로젝트가 포트폴리오에서 강한 이유는 “AWS를 읽는다”가 아니라, 아래를 한 줄로 묶기 때문이다.

- 라이브 인프라 그래프
- 설명 가능한 보안 감사
- 안전 장치를 가진 자연어 질의
- 시간축 기반 변경 추적

즉 Ariadne는 단순한 시각화 장난감이 아니라, **실제 운영자가 묻는 다섯 가지 질문을 오픈소스로 풀어내는 실험**이라고 설명할 수 있다.
